- **[major] Synthetic test sequences produce no interior velocity peak, causing all round-trip tests to fail with SegmentationError** — `tests/vision/test_pipeline.py`
  - The helpers _make_front_kick_sequence and _make_punch_sequence animate a limb with a sin(t) position arc from t=0 to t=π.   - _Fix:_ Rewrite the synthetic sequences to produce a clear interior velocity peak, for example using a Gaussian or (1 - cos(t))/2 position profile so that peak velocity occurs near the middle of the sequence.

## 🐛 Likely bugs & security (10)

- **[blocker] Upload, jobs, and results routers never registered with the FastAPI app** — `backend/main.py`
  - `main.py` only calls `app.include_router(analyze_router)`. The `upload_router`, `jobs_router`, and `results_router` defined in `routers/upload.py`, `routers/jobs.py`, and `routers/results.py` are never included, so every request to `POST /upload`, `GET /jobs/{id}`, `GET /jobs/{id}/results`, and `GET /history` returns HTTP 404. Confirmed by running the upload tests: all 12 upload/jobs tests fail with `assert 404 == 202`.
  - _Fix:_ Import and register the three missing routers in `main.py` with `app.include_router(upload_router)`, `app.include_router(jobs_router)`, and `app.include_router(results_router)`.
- **[blocker] ImportError: `AnalysisResult` does not exist in `backend.models`** — `backend/schemas.py`
  - `schemas.py` imports `AnalysisResult` from `backend.models`, but that ORM module only defines `Upload`, `Job`, `Score`, and `History`. The import fails with `ImportError: cannot import name 'AnalysisResult'`, which also breaks `routers/results.py` and `backend/tests/test_results.py` at collection time. Confirmed by running `python3 -c 'from backend.schemas import AnalysisResultResponse'`.
  - _Fix:_ Either add an `AnalysisResult` ORM model to `models.py` with the fields expected by the schema (`scores`, `metric_deltas`, `keyframe_paths`, `overall_score`, `created_at`, `job` relationship), or rework the schema to use the existing `Score` rows associated with a `Job`.
- **[blocker] ImportError: `align_to_reference` and `score_all_criteria` do not exist** — `backend/scoring/engine.py`
  - engine.py imports `align_to_reference` from `backend.scoring.dtw_aligner` and `score_all_criteria` from `backend.scoring.criteria`, but neither function is defined in those modules. `dtw_aligner.py` only exposes `load_reference_template` and `align_sequence`; `criteria.py` has no aggregator function at all. Confirmed by running `python3 -c 'from backend.scoring.engine import score_rep'` which immediately raises ImportError on the `align_to_reference` import, preventing the entire scoring pipeline from loading.
  - _Fix:_ Add an `align_to_reference(technique, sequence)` wrapper in `dtw_aligner.py` that calls `load_reference_template` then `align_sequence`, and add a `score_all_criteria(technique, aligned_seq)` function in `criteria.py` that dispatches to the six individual scorer functions and returns a `dict[str, tuple[float, float]]`.
- **[major] Extractor uses image-space coordinates (y-down) while scoring assumes world-space (y-up)** — `backend/extractors/landmarks.py`
  - `landmarks.py` extracts `detection.pose_landmarks` which returns normalized image coordinates where y increases downward. All biomechanical scorers in `backend/scoring/criteria.py` (and the module's own docstring) explicitly assume world-space axes with y pointing up. As a result, `chamber_height` measures `np.max(..., axis=1)` on the y channel, but in y-down image space the maximum y is the lowest knee position (not the highest), inverting the score. `balance_com` selects the support foot as the ankle with the lower mean y, which in y-down space is the lifted/kicking foot rather than the planted one. The data-pipeline extractor (`data_pipeline/extractor.py`) correctly uses `pose_world_landmarks`.
  - _Fix:_ Change `backend/extractors/landmarks.py` to use `detection.pose_world_landmarks` instead of `detection.pose_landmarks`, matching the data-pipeline extractor and the coordinate convention expected by the scoring and normalization modules.
- **[major] Job ORM model is missing fields required by schemas and results router** — `backend/models.py`
  - `schemas.py` and `routers/results.py` access `job.job_id` (str), `job.session_id`, `job.technique`, and `job.result` on `Job` ORM instances, but the actual `Job` model only has an integer `id`, `upload_id`, `status`, `error_message`, `created_at`, and `updated_at`. There is no `session_id`, no `technique` column, no string `job_id`, and no `result` relationship (only a `scores` list). Any live call to the results or history endpoints would raise `AttributeError`.
  - _Fix:_ Add `job_id` (a string UUID column), `session_id`, `technique`, and a one-to-one `result` relationship pointing to an `AnalysisResult` model to the `Job` ORM class, and create the corresponding Alembic migration.
- **[major] Criterion names in weights.py do not match names returned by criteria.py** — `backend/scoring/weights.py`
  - `CRITERION_NAMES` and `TECHNIQUE_WEIGHTS` in `weights.py` use `"hip_rotation"` and `"balance"`, but the scorer functions in `criteria.py` emit `CriterionResult` objects with `name="hip_rotation_at_impact"` and `name="balance_com"` respectively. When the scoring engine eventually calls `raw_results[name]` for names from `CRITERION_NAMES`, it would raise `KeyError` for those two criteria, and the per-technique weight dicts would also fail the existing integrity check at import time if corrected names were used.
  - _Fix:_ Align the names: either rename the criterion keys in `TECHNIQUE_WEIGHTS` and `CRITERION_NAMES` to `"hip_rotation_at_impact"` and `"balance_com"`, or change the `name` fields returned by the scorer functions to match the weight keys.
- **[major] `SegmentationError` from `find_rep_window` propagates uncaught out of `preprocess`** — `backend/vision/pipeline.py`
  - `preprocess()` calls `find_rep_window()` without catching `SegmentationError`. Any landmark sequence whose velocity profile has no interior local maximum (e.g., a smooth sinusoidal arc where velocity is highest at the endpoints) causes an unhandled exception that crashes the entire pipeline. All 10 end-to-end pipeline tests fail with this error on the synthetic kick and punch fixtures, confirming the issue is not just a test-fixture flaw but a realistic failure mode for typical motion capture data.
  - _Fix:_ Catch `SegmentationError` in `preprocess()` and either re-raise it as a `ValueError` with a descriptive message so callers can handle it gracefully, or fall back to fixed percentile indices (e.g., frames at 25 %, 50 %, 75 %) when no clear peak is found.
- **[major] `review` and `extract` CLI subcommands missing from the argument parser** — `data_pipeline/cli.py`
  - `cli.py` only registers the `fetch` subcommand in `_build_parser()`, but `data_pipeline/extractor.py` and `data_pipeline/review.py` each implement full `process_clip`/`run_review` entry points that the test suite expects to be reachable as `python -m data_pipeline.cli extract` and `python -m data_pipeline.cli review`. Three tests in `tests/test_review.py` and two tests in `tests/test_extractor.py` confirm this with `SystemExit: 2` / `argument <command>: invalid choice: 'review'`.
  - _Fix:_ Add `extract` and `review` subparser entries to `_build_parser()` with their respective `--landmarks-dir`, `--mp4`, `--reviewer` flags and wire them to `cmd_extract`/`cmd_review` handler functions.
- **[minor] Upload MIME validation relies solely on client-supplied Content-Type header** — `backend/routers/upload.py`
  - The upload endpoint rejects requests whose `file.content_type` does not start with `"video/"`, but the Content-Type value is taken directly from the multipart header set by the client and is never verified against the actual file bytes. A client can upload any file type (including executables or scripts) by simply setting the MIME header to `video/mp4`.
  - _Fix:_ After writing the file, read the first few bytes and validate against known video magic bytes (e.g., `ftyp` for MP4, `RIFF` for AVI), or use a library such as `python-magic` to detect the true MIME type, and reject the upload if the detected type is not a video.
- **[minor] User-controlled `file.filename` may contain path separators, enabling path traversal** — `backend/routers/upload.py`
  - `safe_filename` is constructed as `f"{uuid}_{file.filename}"` and immediately used as `upload_dir / safe_filename`. Python's `pathlib` treats `/` within the right operand of the `/` operator as a path separator, so a filename such as `foo/../../../etc/crontab` splits into path components that can navigate outside the upload directory. If the intermediate directory components happen to exist on the filesystem, the file is written to an attacker-controlled location.
  - _Fix:_ Sanitize the filename before using it in a path: apply `Path(file.filename).name` to strip any directory components, or replace `/` and `..` with safe characters before constructing `safe_filename`.
